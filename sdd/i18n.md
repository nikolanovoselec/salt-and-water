# i18n

Four-locale internationalization with owner-activated languages, content localization, and cultural adaptation.

## Key Concepts

- **Locale**: Language + cultural formatting — hr (Croatian), de (German), sl (Slovenian), en (English)
- **Locale activation**: Owner enables/disables locales from CMS settings
- **Content localization**: Cultural adaptation per market, not word-for-word translation
- **Fallback**: Croatian is the base language, always active, cannot be disabled

## Requirements

### REQ-I18N-1: Locale-Prefixed Routing

- **Intent:** Clean, SEO-friendly multilingual URL structure
- **Applies To:** Visitor, System
- **Acceptance Criteria:**
  - All routes prefixed: `/hr/`, `/de/`, `/sl/`, `/en/`
  - Root `/` redirects to detected locale (Accept-Language header, matching `hr`, `de`, `sl`, `en` — variants like `de-AT` match `de`) or Croatian default
  - Each page has `<link rel="alternate" hreflang="x">` tags **in the HTML `<head>`** for all active locales + `x-default` -> `/hr/`. These are rendered server-side in the page layout, not just in the sitemap.
  - `<html lang="hr|de|sl|en">` set per page
  - Canonical URLs: self-referencing per locale. **Strict URL policy:** no `www`, no trailing slashes, enforced via 301 redirects.
  - Sitemap includes only active locales with `xhtml:link` alternates
  - **Disabled locales:** return 404 with `noindex`. Excluded from sitemap and hreflang.
  - **Legal page exception:** Impressum and Privacy Policy always available in German regardless of DE locale activation (CON-LEGAL). On these pages, language switcher shows DE even if globally disabled. If user switches from a legal DE page to non-legal content while DE is disabled, redirect to HR equivalent with notice.
  - **Fallback policy:** If entire page is unpublished in that locale, return 404.
  - **Partial fallback for published pages:** If a published page in an active locale has missing translated fields:
    - **Rich text fields** (descriptions, page bodies): show Croatian text with page-level banner: "Some content on this page is shown in Croatian because translation is not yet available."
    - **Short structured fields** (titles, amenity labels, button text): silently fall back to Croatian without per-field indicators (preserves layout integrity)
    - **SEO fields / slugs / structured data:** no fallback — use Croatian values internally, not displayed to user
    - **Image alt text:** falls back to Croatian silently (accessibility > language purity)
  - User has locale cookie for disabled locale: redirect to Croatian equivalent with language switcher highlighting available options.
  - Pages available only in subset of locales: hreflang only lists available locales for that page.
- **Constraints:** CON-SEO, CON-I18N
- **Priority:** P0
- **Dependencies:** REQ-I18N-2
- **Verification:** Validate hreflang tags present in HTML `<head>` (not just sitemap), test 404 for disabled locales
- **Status:** Partial — routing works, sitemap has hreflang, `<link rel="alternate" hreflang>` tags now rendered in HTML `<head>` via base layout; disabled-locale 404 behavior and fallback policy not yet verified live

### REQ-I18N-2: Language Activation Settings

- **Intent:** Let owner control which languages are live
- **Applies To:** Owner
- **Acceptance Criteria:**
  - Admin settings: toggle per locale (Croatian always on, greyed out)
  - Activating a locale: routes go live, added to sitemap and language switcher
  - Deactivating: removed from switcher, returns 404, removed from sitemap
  - Warning when deactivating a locale with published content
  - Operable on phone
- **Constraints:** CON-CMS
- **Priority:** P0
- **Dependencies:** REQ-CMS-1
- **Verification:** Toggle locale, verify 404/200 behavior
- **Status:** Planned

### REQ-I18N-3: UI String Translations

- **Intent:** All interface text translated per locale
- **Applies To:** System
- **Acceptance Criteria:**
  - Static UI strings in JSON translation files per locale
  - Covers: nav labels, button text, form labels, error messages, date/number labels, footer text, meta descriptions
  - No hardcoded strings in templates
  - Missing translation falls back to Croatian, logged as warning in dev
  - All UI strings preloaded for all 4 locales at launch
  - **No mixed languages on any single page.** If a page is rendered in locale X, every visible UI string, heading, paragraph, CTA button, and section label must be in locale X. A Croatian heading with an English paragraph or vice versa is a bug. This includes the homepage CTA section, footer, navigation labels, and error messages.
- **Constraints:** CON-I18N
- **Priority:** P0
- **Dependencies:** None
- **Verification:** Switch to each locale, verify no untranslated strings. **Specifically verify the homepage CTA section, contact info, and footer render entirely in the active locale.**
- **Status:** Partial — inline ternaries replaced with t() calls across homepage, listing page, detail page, and zdrelac page; ~20 new translation keys added to all 4 locale JSON files; i18n completeness test added. Mixed-language bug on homepage CTA section addressed. Pending live verification across all locales.

### REQ-I18N-4: CMS Content Localization

- **Intent:** All editorial content translatable per locale independently
- **Applies To:** Owner
- **Acceptance Criteria:**
  - Every content field in Emdash has per-locale variants
  - Owner edits each locale independently from admin (on phone)
  - Locale tabs in content editor — clear which language is being edited
  - Content publishable per locale independently
  - Alt text for images: per locale
  - SEO fields: per locale
  - Missing locale content: page not published in that locale
  - **Publication model:** One global entity per content item (apartment, guide entry, etc.). Localized fields stored per locale. Non-localized fields (pricing, availability, amenity booleans, media) are global — shared across all locales. Publication status is per locale and controls route visibility only. An apartment can be published in HR and draft in DE while sharing the same photos and prices.
- **Constraints:** CON-CMS, CON-I18N
- **Priority:** P0
- **Dependencies:** REQ-CMS-1, REQ-I18N-2
- **Verification:** Create apartment in HR, add DE translation, verify both render
- **Status:** Planned

### REQ-I18N-5: Locale-Aware Formatting

- **Intent:** Dates, numbers, and currencies displayed per locale conventions
- **Applies To:** Visitor
- **Acceptance Criteria:**
  - Dates via `Intl.DateTimeFormat` per locale
  - Currency: EUR with locale formatting via `Intl.NumberFormat`
  - Numbers: locale-appropriate separators
  - Week starts Monday for all locales
- **Constraints:** CON-I18N
- **Priority:** P0
- **Dependencies:** None
- **Verification:** Test display in all 4 locales
- **Status:** Implemented

### REQ-I18N-6: Cultural Content Adaptation

- **Intent:** Content tone and detail level adapted per market
- **Applies To:** Owner, System
- **Acceptance Criteria:**
  - German: precise distances in meters, detailed appliance lists, parking dimensions, exact specs
  - Croatian/Slovenian: warmer, familiar tone, practical logistics focus
  - English: universal clarity, benefit-led descriptions
  - CMS allows different-length descriptions per locale
  - UI labels culturally appropriate (e.g., "Ferienwohnung" not "Apartment" for DE)
- **Constraints:** CON-I18N
- **Priority:** P1
- **Dependencies:** REQ-I18N-4
- **Verification:** Native speaker review per locale
- **Status:** Planned

## Out of Scope

- Auto-translation / machine translation
- RTL language support
- More than 4 locales
- Currency conversion (all prices in EUR)

## Domain Dependencies

- CMS (content schema, admin locale editing)
- Storefront (language switcher)
- SEO (hreflang, sitemap)
- All content domains (apartments, editorial, social proof)
